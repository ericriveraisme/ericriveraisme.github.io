Sovereign Lab Day 2: Evicting Ghosts and Hardening the Fortress

By: Eric | March 11, 2026 | #Proxmox #Networking #CCNA #LearningInPublic #HomeLab

If Day 1 was about the adrenaline of "seeing the lights come on," Day 2 was the sobering reality of maintenance.

I went to bed at the end of Day 1 with a working lab, but I woke up with a "House of Cards." If I rebooted my server, the network fell apart. If I wasn't sitting at my desk to manually type commands, the lab stayed dark. At 40, I’m learning that "working" isn't enough—it has to be persistent.

Here is how I spent Day 2 hardening the Sovereign Lab, battling a "Ghost IP," and finally getting my infrastructure to talk back to me.

🏗️ The "House of Cards" Problem

When I started Day 2, my Core-Router (the brain of the lab) was held together by manual Linux commands. Every time the container restarted, Proxmox would inject a "Ghost IP" (.232) into my interface, clashing with the identity I wanted (.235).

I was a "manual operator," not an architect. To fix this, I had to pivot to enterprise-grade tools.

🛣️ The Cisco Pivot: Entering the World of FRR

Since I am studying for my CCNA, I decided to stop using basic Linux networking hacks and start using FRRouting (FRR).

For the non-networkers: Imagine your router is a busy intersection. Basic Linux commands are like a guy standing in the middle waving his arms. FRR is like installing a fully automated, synchronized traffic light system. It uses a CLI (Command Line Interface) called vtysh that is a near-perfect clone of Cisco’s industrial standards.

The "Golden Config"

Instead of messy scripts, I "burned" a configuration into the router's memory. Now, my routing table isn't a guess—it’s a set of rules.

! My Cisco-style Router Config
interface eth0
 description WAN-Link-to-Home-Router
 ip address 192.168.0.235/24
!
interface eth1.10
 description Management-VLAN-10
 ip address 10.0.10.1/24
!
ip route 0.0.0.0/0 192.168.0.1

👻 Exorcising the Ghost IP

Even with FRR, the Proxmox host kept trying to force that .232 IP onto my router. It was a classic "Source of Truth" conflict.

The Lesson: Sometimes, you can't just ask a service to stop; you have to "evict" the problem. I wrote a startup script that runs the moment the router wakes up. It waits two seconds, deletes the "Ghost" IP, and restarts the routing service.

The Fix (/usr/local/bin/evict-ghost-ip.sh):

#!/bin/bash
# 1. Wait for the ghost to appear
sleep 2
# 2. Evict the squatter
ip addr del 192.168.0.232/24 dev eth0 2>/dev/null
# 3. Hand the keys to the professional (FRR)
systemctl restart frr

📮 The "Silent Sentry": Solving the SMTP Gauntlet

I wanted my lab to email me backup reports. This sounds simple, but it was the "Final Boss" of Day 2. I was fighting Postfix (a mail server) and Google’s security layers.

My Biggest "Rookie" Mistake

I spent hours staring at error logs that said 535 5.7.8 Authentication Failed. I checked my passwords, my encryption, and my firewall.

The Moment of Humility: I realized I had left a placeholder email (your-email@gmail.com) in my configuration file instead of my actual address. I had built a high-tech rocket ship but forgot to tell it who the pilot was.

The Lesson: At the end of a long shift, always check your placeholders. As soon as I updated the email and forced the mail server to use IPv4 (to avoid residential ISP blocks), the logs turned green.

🛡️ Tier 3 Redundancy: Making it Native

Finally, I moved away from brittle backup scripts. I used Proxmox’s native CIFS (SMB) integration to "mount" my workstation's 14TB external drive directly into the datacenter.

Now, the Proxmox GUI handles the backups. At 3:00 AM, it snaps my VMs and sends them across the mesh network (Tailscale) to my physical P: drive. No scripts, no manual labor—just enterprise orchestration.

🏆 Current Standing: Where we are now

  2. Automated Governance: I get an email every morning confirming my data is safe.
  3. Global Command: I can power the entire lab up or down with a single command (lab_up / lab_down) from the Proxmox shell.
